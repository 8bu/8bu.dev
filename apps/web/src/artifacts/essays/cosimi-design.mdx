This chat is powered by Cosimi. There is no LLM in the request path. Replies come from a curated pair store, scored by a three-tier matching engine that runs entirely against Postgres and a small TypeScript matcher. The corpus is hand-written. The voice is mine. The system either has an answer I wrote, or it tells you it does not.

This essay is a design write-up of why the chatbot is shaped that way and what the trade-offs cost.

## The problem (why a chatbot without an LLM)

The default move for a portfolio chat in 2026 is to bolt a hosted LLM onto a "talk to me" surface and call it done. That solves the dialogue problem - the model will reply to anything - but it creates a worse problem. The replies are no longer mine. They are the model's prior over what a developer named Long might plausibly say, smoothed over with corporate-safe verbs.

If a recruiter asks me a question I have not thought through, I would rather the answer be silence than a confident hallucination in my voice. The cost of a wrong-shaped reply is higher than the cost of "I do not have a pattern for that yet, but here is how to reach me." Pattern matching makes the failure mode legible: either there is a curated reply, or there is not.

There is a second reason. The corpus IS the writing. Drafting deflections and process notes and identity statements as YAML seeds forced me to actually have positions on things. An LLM shortcut would have let me skip that work.

## The matcher (three tiers explained)

User input flows through three tiers in order. The first tier to return a non-null result wins. No tier escalation if a higher tier scores poorly enough.

**Tier one: exact match.** A normalized version of the input (NFC, lowercased, whitespace-collapsed) is compared against an accent-stripped form of every stored pattern. Hits return in single-digit milliseconds. This tier carries the "hello", "what stack", "what is your rate" class of inputs - patterns I expect to see verbatim.

**Tier two: Postgres full-text search.** The input gets converted to a `tsquery`, run against a GIN-indexed `tsvector` column on the patterns table, and ranked with `ts_rank`. The rank is clamped to `[0, 1]` and compared against a configurable threshold (`MATCH_FTS_MIN`). This tier handles word-order variants and partial matches - "do you take freelance work" matching a pattern stored as "freelance availability".

**Tier three: trigram similarity.** Postgres's `pg_trgm` extension splits both sides into character trigrams and computes a similarity score. The query uses the `%` operator (which is GIST-indexed and fast) AND an explicit `similarity(...) >= MATCH_TRGM_MIN` filter. The double filter matters - `%` reads from a process-global GUC that is mutable, so pinning the threshold explicitly is how the behavior stays stable across deploys.

After ordering by tier, the matcher takes the top K (default 5) and picks one at random in JS. That randomness is intentional. A chat that always returns the same reply for the same input feels mechanical. A chat that picks among a few hand-written variants feels alive without lying about being alive.

The cascade has a fourth tier in practice: session-scoped teaches. If you taught the bot a reply earlier in the session via `/teach`, that reply gets first-look priority for ten minutes. After that the teach falls out of the session cache and either gets promoted to the curated corpus by me, or quietly expires.

## The corpus discipline

The pair store is the system. Every curated reply lives in `seeds/portf/*.yaml` files - one per topic. `identity.yaml` for who I am. `stack.yaml` for what I work with. `process.yaml` for how I work. `deflections.yaml` for the questions I do not want to answer on a public chat surface. `hiring.yaml` for recruiter-shaped inputs.

Editing the corpus is a real act of writing. Each pattern has a normalized input and a response. Each response has to read like something I would actually say - terse, technical, no marketing-speak. No em-dashes. En-dashes only in date ranges. Concrete claims only.

The seeds run through a CLI that flattens YAML into `pairs` table rows via an `insertManyPairs` write path. Every seed run creates one `import_batches` row, which IS the unit of rollback. Re-seeding creates duplicates by design - the assumption is that if you re-ran a seed file, you wanted those rows again.

## Trade-offs (depth-for-honesty)

The system cannot reply to inputs it has not seen patterns for. That is the point.

The cost is real. A recruiter asking about my experience with, say, a niche library I have never worked with will get a fallback response. The fallback says something like "I do not have a pattern for that yet" and links a contact path. That is honest but it is also a worse experience than an LLM that would have improvised a plausible answer.

The trade is depth-for-honesty: every reply is a reply I wrote. I would rather be slow than fluent in someone else's voice. For a portfolio chat that is the right trade. For a customer-support bot, it probably is not. Cosimi is not a general-purpose chatbot engine - it is a stance about what the chatbot's voice should be, with code attached.

## Implementation notes (monorepo shape, postgres FTS + trigram, no LLM at runtime)

pnpm + Turbo monorepo. Four apps, eight shared packages.

- `apps/api` is the public Hono REST + SSE server. Binds `PORT` on `0.0.0.0`. No LLM imports anywhere in its dependency graph.
- `apps/admin-api` is a separate Hono process bound to `127.0.0.1`. Process split + network-layer gate IS the auth contract - there is no app-level auth on admin routes.
- `apps/web` is the public chat UI (Vite + React + Tailwind v4). `apps/admin` is the internal dashboard.
- `packages/matcher` is the three-tier engine. Pure TypeScript over a postgres.js client.
- `packages/normalizer` is the NFC + lowercase + whitespace step. It deliberately preserves diacritics. The accent-stripping happens server-side via a `f_unaccent` SQL function. Splitting normalization across language and database is a discipline thing - the JS layer cannot accidentally desync from the SQL layer.

Streaming replies use SSE. The client receives token events at a paced rate (`SSE_DELAY_*`) so the reply feels like it is being typed rather than dumped. The pacing is the only part of the system that pretends to be an LLM.

## What it cost (gaps the system cannot fill)

The honest gaps:

- Long-tail inputs miss. If you phrase a question in a way no pattern covers and no trigram similarity catches, you get the fallback.
- The corpus needs maintenance. Every couple of weeks I read the `unanswered` table in the admin app, write new patterns, retire stale ones.
- Onboarding new topics is slow. Adding "what I think about X" means actually thinking about X and writing 3-10 variant patterns.

That last cost is the one I underestimated. The system pushed the work back onto me. That is the cost of not having an LLM.

## What I'd do differently

Two things.

First, the locale story. Pairs carry a locale tag (`vi`, `en`, or `und`). The cascade includes both the requested locale and `und` for fallback coverage. The split helped early but the `und` bucket grew into a junk drawer. Next iteration I would split `und` into "intentionally universal" vs "untagged legacy" and tighten the cascade.

Second, the session teach TTL. Ten minutes is short enough that interesting teaches expire before I see them. I would extend the TTL to a day and add an explicit "promote to corpus" admin flow. The point of teaches was always to harvest them, not to keep them session-bound.

The system is shipping. The matcher is fast. The corpus is mine. That is what I wanted.
